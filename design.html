Building a Natural Language Data Agent: Design & Architecture

How we designed a local AI data agent inspired by OpenAI's approach, powered by Groq and Llama 3.3

What if anyone on your team could ask a data question in plain English and get an accurate, well-reasoned answer in seconds — without writing a single line of SQL?

That's the promise of AI data agents. In January 2026, OpenAI published a detailed blog about their in-house data agent that does exactly this across 600 petabytes of internal data. We took their ideas and built a fully functional local demo: a Python-based data agent that uses Groq's Llama 3.3 70B to translate natural language questions into SQL, execute them against a local SQLite database, and present synthesized answers.

In this 4-part series, we'll walk through the entire journey. This first post covers the why, the what, and the how of the architecture.

The Inspiration

From OpenAI's Blog

"Our data agent lets employees go from question to insight in minutes, not days. This lowers the bar to pulling data and nuanced analysis across all functions, not just by our data team."

OpenAI's agent operates across 70,000 datasets, uses GPT-5 for reasoning, and features six layers of context enrichment including code-level table understanding, institutional knowledge from Slack and Docs, and a self-learning memory system. It's sophisticated — and built for their internal scale.

Our goal was different: build a minimal, self-contained demo that captures the core concepts anyone can run locally. Specifically:

  • A realistic business use case with 4-5 related tables
  • Context-grounded SQL generation (not naive text-to-SQL)
  • A self-correction loop that fixes broken queries
  • A learning system that remembers past mistakes
  • A conversational interface that supports follow-up questions

The Business Use Case: CloudMetrics

We chose a B2B SaaS subscription analytics scenario — a company called "CloudMetrics" that sells project management software. This domain works well because it naturally involves:

  • Time-series data — subscriptions, invoices, and tickets all have date dimensions
  • Multi-table joins — answering "revenue by plan" requires joining invoices → subscriptions → plans
  • Business logic — "MRR" means only active subscriptions; "revenue" means only paid invoices
  • Non-trivial queries — churn rates, resolution times, and ranked results require CTEs and window functions

The database has 5 tables with realistic relationships:

Architecture: The 4-Layer Context Model

The single biggest lesson from OpenAI's blog is that context is everything. A raw LLM given just "What is our MRR?" and a database schema will generate plausible-looking but often incorrect SQL. It might include churned subscriptions, use PostgreSQL syntax on SQLite, or join on the wrong columns.

Our agent uses four layers of context, simplified from OpenAI's six, that are assembled into every LLM call:

Layer 1: Schema Context

Complete table metadata — column names, types, descriptions, primary keys, foreign key relationships. Stored in schema.json.

Layer 2: Business Rules

Metric definitions, calculation formulas, and "gotchas" — things like "use julianday() not EXTRACT in SQLite". Stored in business_rules.json.

Layer 3: Query Patterns

Pre-validated SQL examples annotated with keywords. Retrieved by matching against the user's question. Stored in query_patterns.sql.

Layer 4: Learnings

Auto-accumulated corrections from past self-corrections. The agent saves what went wrong and how it was fixed, and avoids the same mistake next time.

These layers are not equally important for every query. For a simple "How many customers do we have?", the schema context alone is sufficient. For "What is our monthly churn rate?", the business rules layer is critical to get the definition right. For date-related queries, the SQLite gotchas prevent the most common class of errors.

Key Design Decisions

This demo makes two deliberate simplifications. Both are explicitly designed to be swappable for production-grade alternatives.

LLM: Groq + Llama 3.3 70B

Fast 128K context Free tier

Groq provides extremely fast inference on open models. The Llama 3.3 70B model is capable enough for SQL generation while being significantly cheaper than proprietary alternatives. Its 128K context window comfortably fits our schema, business rules, query patterns, and conversation history in every call.

Production upgrade: Swap to GPT-4, Claude, or a fine-tuned domain-specific model. The llm_client.py interface stays the same — only the API endpoint and model name change.

Retrieval: Keyword Matching

Simple 8 patterns

For a demo with 8 query patterns and a handful of learnings, simple set intersection on tokenized words is sufficient. Each query pattern is annotated with keywords; when a user asks a question, we tokenize it and find patterns with the most keyword overlap. No external dependencies, no model loading, instant results.

Production upgrade: Use embedding-based retrieval with a vector database (e.g., Pinecone, Chroma, or pgvector). Embed all query patterns and learnings, then retrieve by cosine similarity. This scales to thousands of patterns and handles semantic matching far better than keyword overlap.

Project Structure

dataagent/
+-- requirements.txt          # groq, faker, rich, python-dotenv
+-- .env.example              # GROQ_API_KEY template
+-- setup_db.py               # Creates SQLite DB + synthetic data
+-- app.py                    # Main CLI REPL
+-- knowledge/
|   +-- schema.json           # Layer 1
|   +-- business_rules.json   # Layer 2
|   +-- query_patterns.sql    # Layer 3
|   +-- learnings.json        # Layer 4 (starts empty)
+-- agent/
    +-- llm_client.py         # Groq API wrapper
    +-- context_builder.py    # Assembles 4 layers into prompt
    +-- sql_executor.py       # SQLite execution + formatting
    +-- self_correction.py    # Retry loop + learning save
    +-- conversation.py       # History management
    +-- response_parser.py    # Extracts SQL from LLM output

The separation between knowledge/ (static context files) and agent/ (runtime logic) mirrors OpenAI's distinction between offline context enrichment and the runtime agent. The knowledge files can be edited independently of the code — improving them directly improves the agent's output without changing a line of Python.
